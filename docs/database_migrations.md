# Database migrations

The local SQLite database is `chess_master.sqlite3`. Schema version 2 is defined
as reviewed SQL in `lib/core/database/database_schema.dart`.

## Version 1 invariants

- Foreign-key enforcement is enabled on every connection.
- Initial creation runs in one transaction.
- IDs are stable text values generated by their owning domain feature.
- Times are UTC milliseconds since the Unix epoch; a challenge's local calendar
  date is stored separately as `YYYY-MM-DD`.
- Booleans use constrained integer values.
- JSON columns contain data only. Imported values are parsed and validated; they
  are never evaluated or executed.
- Moves are unique by `(game_id, ply)` and cascade with their game.
- One reward source cannot create duplicate transactions.
- A claimed challenge, achievement, or tutorial reward requires completion first.
- Android system backup is disabled; explicit versioned export is the supported
  backup path once implemented.

## Adding a migration

1. Increase `DatabaseSchema.currentVersion` by exactly one.
2. Add immutable SQL statements for that version.
3. Run them inside the existing upgrade transaction.
4. Record start/completion/failure in `data_migrations` without sensitive data.
5. Add tests for empty creation, upgrade from every supported old version,
   interrupted writes, constraints, serialization, and retained data.
6. Update the backup compatibility matrix and this document.

Never edit an already released migration in place. A failing migration must roll
back atomically and leave the previous schema usable. Downgrades are rejected;
automatic destructive recreation is prohibited.

## Backup envelope (planned)

The future export format will contain schema version, app version, export time,
selected data sections, checksum, and compatibility metadata. A preview and
explicit confirmation are required before replacement. Unknown fields may be
ignored only when the format version says they are forward-compatible; unknown
executable content is never accepted.

## Version 2

Version 2 adds:

- explicit non-negative `coin` and `hint` wallet rows;
- a rebuilt reward ledger with before/after balances, application version,
  related challenge, stable sequence, and chained integrity fields;
- separate coin and hint amounts on daily challenge definitions;
- deduplicated challenge-event receipts.

The upgrade copies v1 ledger rows, reconstructs their prior balance, retains
challenge foreign keys, seeds each wallet from its latest retained entry, and
marks legacy integrity values as migration anchors. New entries chain from that
anchor. Fresh installs execute v1 and v2 SQL in the same creation transaction.

Unit tests execute both clean creation and v1-to-v2 upgrade against real SQLite
through `sqflite_common_ffi`, validate foreign keys and columns, retain legacy
challenge/reward rows, and exercise concurrent claims.

No general application-data import UI exists yet. Reward-ledger JSON copy is a
read-only export and cannot be imported or executed.
